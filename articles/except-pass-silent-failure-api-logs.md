ジョブは毎日「成功」と出ていました。実際には、22種類のうち14種類の取得が毎回失敗していました。その状態が、約1ヶ月続いていました。

自宅ラボで外部APIからデータを集めていて、いちばん怖い失敗は「落ちた」ではありません。落ちたなら、まだ見つけられます。厄介なのは、ジョブ全体は成功扱いで、ログにも成功と出ていて、実際には中の取得だけが失敗し続けている状態です。原因は監視基盤の不足でもリトライ回数でもなく、`except: pass` の1行でした。

## 何が起きていたか

対象は、バイタル系の健康データを外部APIから2時間おきに収集するノードです。要求していた取得対象は22種類。ジョブの終了コードは常に正常で、成功ログだけが積み上がっていました。

失敗種別を出すログを1行だけ足した直後の実行で、内訳が見えました。

| 分類 | 件数 | 観測された意味 |
| --- | --- | --- |
| 認可スコープ外・未記録の種別 | 7種 | APIは403を返していた |
| データソース自体が存在しない | 6種 | APIは400を返していた |
| 一時障害 | 1種 | APIは503を返していた |

重要なのは、14件の失敗がすべて「サーバーが不安定だった」わけではない点です。一時障害は1種だけで、大半は、そもそもそのアカウントで取れないもの、またはデータソースとして存在しないものを、こちらが毎回頼み続けていた結果でした。APIは正しくエラーを返していました。間違っていたのは、そのエラーを握り潰して成功に見せていた側です。

## `except: pass` が消すのは処理ではなく失敗数

`except: pass` は、書いた瞬間には便利に見えます。1種類欠けただけで全体を止めたくない、外部APIの一時的な揺れで毎回通知したくない。その判断自体は理解できます。

ただし、問題は「処理を続けること」ではありません。「何が何回失敗したか」を消すことです。設計として並べると差がはっきりします。

| 設計 | メリット | デメリット |
| --- | --- | --- |
| 例外で即停止 | 失敗に気づきやすい | 一部の欠損で全体が止まる |
| `except: pass` で継続 | ジョブは止まりにくい | サイレント失敗になりやすい |
| 失敗を数えて継続 | 欠損を見える化しつつ処理を続けられる | ログ設計が少し必要 |

自宅ラボでは、常時人が画面を見ていません。無人運用を前提にするなら、成功ログではなく失敗数が残る形に寄せる必要があります。「実験中の一時コードに限って握り潰す」なら成立しますが、定期実行の収集系に残すものではありませんでした。

## 追加したのはログ1行だけ

最初から大きな監視基盤を足したわけではありません。失敗種別を出すログを1行追加しただけです。

```python
try:
    collect_one_metric(metric)
except ApiError as error:
    logger.warning("metric_collect_failed metric=%s status=%s", metric.name, error.status_code)
```

実運用では、API名・アカウント情報・内部ノード名・認証情報を出さないよう注意します。公開できるログと、運用者だけが見るログは分けて扱うべきです。

自分のコードベースを点検するなら、まず握り潰しを数えます。

```console
$ rg "except.*pass"
```

見つかった箇所をすべて即座に消せ、という話ではありません。例外を無視してよい場面はあります。ただし、**定期実行・外部API・データ収集**の3条件が重なる場所では、少なくとも失敗種別と件数は残したほうがいいです。

## 修正は「現実に存在する9種へ絞る」だった

調査の結果、アカウントに実在するデータソースは11種でした。そのうち、実際に必要な取得対象を9種に絞りました。

| 段階 | 状態 |
| --- | --- |
| 変更前 | 22種を要求 |
| 観測追加直後 | 14種が毎回失敗していると判明 |
| 調査後 | 実在するデータソースは11種 |
| 修正後 | 必要な9種に絞り、エラー0 |

必要なデータは取れたまま、エラーだけが消えました。エラーを減らす最良の方法は、リトライを増やすことではなく、取れないものを頼むのをやめることでした。要求リストが現実とずれていた、というだけの話です。

死活監視は必要ですが、死活監視だけでは「中身の取得が失敗している」は拾えません。ジョブ全体の成功・失敗と、取得対象ごとの成功・失敗は、別の指標として持つ必要があります。バックアップで成功ログだけを見ても復元できるとは限らないのと、同じ構造です。

## 再発防止チェックリスト

1. `except: pass` または広すぎる例外捕捉を検索する。
2. 外部API呼び出しの失敗種別を1行だけログに出す。
3. ジョブ全体の成功と、取得対象ごとの成功を分ける。
4. 403、400、503を同じ失敗として扱わない。
5. 取得対象リストが、実在するデータソースと一致しているか棚卸しする。
6. 取れないものを、毎回取りに行かない。

この順番なら、監視基盤を増やす前にかなりの沈黙を剥がせます。

今回の数字は、22種要求、14種失敗、不可視期間は約1ヶ月、実在データソースは11種、必要な9種へ絞ってエラー0です。「動いているように見える」と「動いている」の差は、握り潰しの1行でした。監視は状態ではなく、失敗を数えたほうがいいです。そしてAPIエラー握り潰しの対策として最初に見るべきものは、通知設定でもリトライ回数でもなく、そもそも取れないものを頼んでいないか、です。
